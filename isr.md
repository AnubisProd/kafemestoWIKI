# **ISR (Incremental Static Regeneration)** в Nuxt 3.

> Это один из самых мощных и популярных гибридных режимов рендеринга в 2025–2026 годах.
Он позволяет получить почти скорость **статического сайта** (SSG),
но при этом обновлять контент без полного перестроения (rebuild) всего проекта.



### Кратко: что такое ISR в двух словах
- Первый посетитель → страница генерируется на сервере → сохраняется как статический HTML + _payload.json
- Все последующие посетители → получают **готовый кэшированный** HTML из CDN (очень быстро, часто < 50–100 мс TTFB)
- Через заданное время (или при on-demand триггере) → страница **перегенерируется в фоне** (stale-while-revalidate)
- Пользователь всегда видит **не устаревшую** версию, даже если регенерация ещё идёт

Это классическая схема **stale-while-revalidate** + кэширование на уровне CDN (Vercel, Netlify и др.).

### Сравнение режимов рендеринга в Nuxt 3

| Режим          | Когда генерируется HTML | Скорость первого запроса | Скорость последующих | Обновление контента          | Подходит для твоего кафе-сайта? |
|----------------|--------------------------|---------------------------|-----------------------|------------------------------|---------------------------------|
| SSR            | на каждый запрос         | средняя (200–800 мс)      | средняя               | мгновенное                   | да, но медленнее на холодном старте |
| SSG (nuxt generate) | на этапе `nuxt build` | очень быстро (<100 мс)   | очень быстро          | только новый деплой          | да, если меню редко меняется |
| ISR            | первый запрос + фоновая регенерация | быстро после первого     | очень быстро (CDN)    | через интервал / on-demand   | **идеально** для большинства страниц кафе |
| SWR (stale-while-revalidate) | как ISR, но без CDN-кэша | быстро после первого     | быстро                | фоновая регенерация          | когда CDN не нужен / локальный деплой |
| CSR (SPA)      | в браузере               | медленно (JS download)    | быстро после загрузки | мгновенное (API)             | нет, плохой SEO |

### Как включить ISR в Nuxt 3 (самый важный блок)

Всё делается через **`routeRules`** в `nuxt.config.ts`:

```ts
export default defineNuxtConfig({
  routeRules: {
    // Вариант 1: ISR с TTL (рекомендую для большинства случаев)
    '/':              { isr: 600 },     // обновлять каждые 10 минут
    '/menu':          { isr: 3600 },    // каждые 60 минут — меню меняется редко
    '/about':         { isr: 86400 },   // раз в сутки

    // Вариант 2: ISR без жёсткого TTL (регенерация только при on-demand или деплое)
    '/contact':       { isr: true },

    // Вариант 3: всё остальное — чистый SSR
    '/api/**':        { cache: false }, // или просто не указывать
    '/admin/**':      { ssr: true },    // всегда свежий SSR

    // Глобально для динамических маршрутов (если будет /blog/[slug])
    '/blog/**':       { isr: 1800 },    // 30 минут
  },

  // Опционально: чтобы Nitro лучше понимал, что мы на Vercel
  nitro: {
    preset: 'vercel'
  }
})
```

- `isr: number` → максимальный возраст кэша в секундах (после этого — фоновая регенерация)
- `isr: true` → кэш навсегда, пока не придёт запрос на регенерацию (on-demand) или не задеплоишь заново

### Как работает на практике (последовательность)

1. Пользователь заходит на `/menu`
2. Если страница уже сгенерирована и кэш свежий → отдаётся HTML из CDN → 20–80 мс
3. Если кэш истёк (или первый запрос) → Nitro генерирует страницу → отдаёт свежий HTML пользователю
4. **В фоне** запускается регенерация → новый HTML + _payload.json сохраняется в кэш
5. Следующий пользователь → уже получает обновлённую версию

### Важные нюансы в Nuxt 3 (2025–2026)

- Работает только на платформах с поддержкой **on-demand ISR** и CDN-кэша: **Vercel**, **Netlify** (частично), **Cloudflare Pages** (через workers)
- Генерируется **_payload.json** рядом с HTML → ускоряет клиентскую навигацию (Nuxt берёт данные из payload, а не делает fetch заново)
- Для динамических маршрутов (`/[slug]`) используй glob: `'/products/**': { isr: 3600 }`
- On-demand регенерация (через API) пока не встроена нативно, но можно сделать через webhook → `fetch('/api/revalidate?secret=xxx')` → `useNitroApp().hooks.callHook('nitro:cache:clear')`
- ISR **не работает** локально в `nuxt dev` — только после `nuxt build` + деплой

### Рекомендация именно для твоего проекта (кафе в Ялте)

```ts
routeRules: {
  '/':              { isr: 600 },     // главная с hero — обновлять часто
  '/menu':          { isr: 7200 },    // меню — раз в 2 часа хватит
  '/about':         { isr: 86400 },   // о нас — раз в день
  '/**':            { isr: 3600 },    // всё остальное (если добавишь страницы)
}
```

`nuxt build` → `git push` → Vercel сам подхватит и будет использовать ISR.

Если меню будет приходить из CMS (Strapi, Contentful, Sanity и т.д.) — ISR + webhook на обновление контента = почти идеальное решение: скорость статического сайта + свежесть данных.
